# Testing Fixtures — Gmail Accounts & Dummy Files

Real accounts and realistic-looking-but-entirely-synthetic data for dev + CI. **No real PII. Ever.**

## 1. Gmail Test Accounts

Create **three** disposable Google accounts. All credentials live in `.env.test` (gitignored) and, in CI, in GitHub Actions Secrets.

| Account | Purpose | Credentials env var |
|---|---|---|
| `shieldme.dev.sender@gmail.com` | Drafts outgoing emails; trigger Email Guardian | `TEST_GMAIL_SENDER_USER`, `TEST_GMAIL_SENDER_PASS` |
| `shieldme.dev.recipient@gmail.com` | External recipient for "unknown domain" path | `TEST_GMAIL_RECIPIENT_USER`, `TEST_GMAIL_RECIPIENT_PASS` |
| `shieldme.dev.drive@gmail.com` | Owns the Drive test corpus for Module 4 | `TEST_GMAIL_DRIVE_USER`, `TEST_GMAIL_DRIVE_PASS` |

### 1.1 Account Setup Checklist

For each account:
1. Create at `accounts.google.com/signup`. Use a throwaway phone number if required (Google often prompts; try desktop first, switch to mobile UA if blocked).
2. Enable 2FA with an authenticator app — store the **TOTP seed** in the secrets manager (app password flow is deprecated; Playwright uses TOTP).
3. Disable the "low security app" / interstitial screens by logging in once on Chrome with a real user agent.
4. For the Drive account: create a top-level folder `ShieldMe Test Corpus` and share a subfolder **"Public (anyone with link)"** with the link-anyone permission. Populate with the PII-bearing fixtures below.

### 1.2 Playwright auth state

Playwright project `auth.setup.ts` logs in once per account, saves `storageState` to `tests/e2e/.auth/*.json` (gitignored). E2E tests inject the state. TOTP handled by a helper using `otplib`.

### 1.3 OAuth scope for Drive testing

See [quickstart.md §5](../specs/001-shieldme-mvp/quickstart.md). Add all three test accounts as **test users** on the OAuth consent screen — they bypass the unverified-app warning.

### 1.4 Hygiene

- Empty the Drive trash and Gmail sent folder weekly (cron in `scripts/clean-test-accounts.mjs`).
- Rotate passwords every 90 days.
- If any account is locked: document recovery in `docs/ops/test-account-recovery.md` (not yet written — blocker is lived experience, add after first lockout).

### 2.7 Generation Recipes

Each recipe (`scripts/fixtures/recipes/*.ts`) takes a Faker seed and produces a file. Example outline:

```ts
// scripts/fixtures/recipes/tax-return-2025.ts
export default defineRecipe({
  seed: 4711,
  output: "tests/fixtures/samples/tax-return-2025.pdf",
  build: (faker) => {
    const doc = new jsPDF();
    doc.text("ΦΟΡΟΛΟΓΙΚΗ ΔΗΛΩΣΗ 2025", 10, 10);
    doc.text(`ΑΦΜ: ${syntheticAfm(faker)}`, 10, 20);       // valid checksum
    doc.text(`Όνομα: ${faker.person.fullName()}`, 10, 30);
    doc.text(`Διεύθυνση: ${faker.location.streetAddress()}, ${faker.location.city()}`, 10, 40);
    doc.addPage();
    doc.text(`IBAN: ${syntheticIban("GR", faker)}`, 10, 10); // valid mod-97
    return doc.output("arraybuffer");
  }
});
```

**Every** synthetic identifier passes its validator (so detectors find them) but is guaranteed not to collide with any real human's identifier (we use documented invalid ranges where possible, else astronomically-unlikely Faker seeds).

### 2.8 Corpus files (separate from samples)

`tests/fixtures/corpus/<country>/<detector>/{positive,negative}.txt` — one candidate per line. These are **text lines**, not files. Positives must pass the detector (with validator where applicable); negatives must not. ≥20 positives, ≥20 negatives, per detector, per country that detector claims to cover.

Corpus generation is **manual-review first, script-padded second**: engineering reviews the initial 20 per country; Faker-padded adversarial negatives extend the negative set.

## 3. Anti-Leak Rules (for this repo)

1. **Zero real PII ever committed.** A pre-commit hook (`scripts/pre-commit-pii-scan.mjs`) runs the detection engine on every staged file. Any finding blocks the commit.
2. **No credentials in the repo.** GitHub push protection + gitleaks in CI.
3. **Test accounts have no real personal info.** Names/DOBs on these accounts are fake. Recovery email is a shared team inbox, not personal.
4. **Fixture PDFs carry a "synthetic — ShieldMe test data" watermark** rendered in the margin so an accidentally-leaked fixture is obviously fake.

## 5. Manual Smoke Checklist (pre-release)

Before tagging a release, a human runs this in ≤15 min:

- [ ] Load unpacked build. Complete first-run flow in ≤5 clicks.
- [ ] Drop `tax-return-2025.pdf` → findings match expected.
- [ ] Drop `big-doc-15mb.pdf` → Free-tier upsell shown.
- [ ] Open Gmail compose, paste IBAN, click Send → modal shown; Go Back works; Send Anyway sends.
- [ ] Break Gmail selectors deliberately (DevTools DOM hack) → banner appears within 1 s.
- [ ] Connect Drive (test account) → audit runs; top-5 critical visible on Free.
- [ ] Password check with `password123` → ≥1 breach.
- [ ] Data broker checklist → mark one site complete → persists across popup reload.
- [ ] Settings → Delete all my data → confirm dialog → all state wiped.

If any step fails, block release.
